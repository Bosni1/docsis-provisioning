  1  """ 
  2  The heart of the database abstraction layer. It defines the L{Record} class 
  3  which represents the database at the table row level. 
  4   
  5  The Record objects are designed to work with tables of a very specific structure. Current  
  6  implementation involves heavy usage of PostgreSQL features such as table-inheritance and 
  7  C{SELECT ... FROM ONLY ...} . 
  8   
  9  The assumptions may be roughly summarized as follows: 
 10   
 11     1. All tables are subtables (subclasses, descendants) of a table named C{object} with this 
 12     signature: 
 13        - objectid : int8 PRIMARY KEY 
 14        - objecttype : varchar (I{name of the actual table}) 
 15        - objectcreation, objectmodification : timestamp 
 16        - objectscope : int2 
 17     2. All foreign keys reference the C{objectid} column in the referenced table. 
 18     3. Multiple-column references are not supported (simplicity). 
 19      
 20  I{for more specific information on the database structure refer to the database docs}       
 21   
 22  Usage 
 23  ===== 
 24   
 25  Recommended usage pattern of the Record class: 
 26   
 27     >>> newRecord = Record.EMPTY ( "customer" ) 
 28     >>> newRecord.objectid 
 29     None 
 30     >>> newRecord.email = "customer@server.com" 
 31     >>> newRecord.telephone = "001 123 445 444" 
 32     >>> newRecord.write() 
 33     >>> newRecord.objectid 
 34     1554 
 35     >>> oldRecord = Record.ID ( 1554 ) 
 36     >>> oldRecord.ofTable ( "customer" ) 
 37     True 
 38     >>> oldRecord.email = "customer@customers.com" 
 39     >>> oldRecord.isModified 
 40     True 
 41     >>> oldRecord.email 
 42     'customer@customers.com' 
 43     >>> oldRecord.read() 
 44     >>> oldRecord.isModified 
 45     False 
 46     >>> oldRecord.email 
 47     'customer@server.com' 
 48     >>> oldRecord.setObjectID(1667) 
 49     >>> oldRecord.email 
 50     'othercustomer@server.com' 
 51     >>> oldRecord.delete() 
 52

 72      Attributes 
 73      ========== 
 74       
 75      A Record object is a object-mapping of rows of database data. The API allows accessing 
 76      column values in one og two convenient ways: 
 77       
 78      C{aRecord.I{columnname}} 
 79       
 80      C{aRecord["I{columnname}"]} 
 81       
 82       
 83      A Record object is set up using the meta-data in a L{Table<ProvCon.dbui.meta.table.Table>} object. 
 84       
 85      All internal field names start with an '_' underscore character to avoid name clashes with 
 86      column names. Column names that start with an underscore will not  be accessible. 
 87           
 88      The getattr and setattr methods are overridden to provide the following functionality: 
 89         1. changing the _objectid attribute causes the object to be read from the database (this 
 90         is wrapped be the setObjectID method. 
 91          
 92         2. changing the _table attribute causes the record to be reset as an empty record 
 93         of the newly set table 
 94          
 95         3. special attributes PP_* can be used to get a nice textual representation of the record              
 96             - PP_TABLE prints a tabular representation of the record. 
 97              
 98         4. attributes named <field_name>_REF return the Record referenced by the field. 
 99              
100             >>> aCustomer.location_REF  
101             aLocationRecordObject 
102       
103      References 
104      ========== 
105       
106      There are three modes in which referencing columns may be treated: 
107         1. B{none}: do nothing, leave the objectid as the column value 
108         2. B{text}: I{columnname}_REF returns a textual representation of the referenced record 
109         3. B{record}: I{columnname}_REF holds the actual referenced record 
110          
111      Record.__default_reference_mode__ is the global default mode of resolving references. 
112      Object-specific mode may be passed to the record constructor with the C{resolvereference} 
113      keyword argument. 
114       
115      objectid, _objectid, setobjectID 
116      ================================ 
117       
118      Perhaps the most confusing part of the Record object is the way 'objectid' is handled. 
119       
120      The _objectid attribute always holds the current objectid, and setting it to a value triggers 
121      one of two actions: 
122         - if the value is not None, and is different from the current value, the record is read 
123         from the database, 
124         - if the value is None, the record is reset as a new record (isNew returns True) 
125       
126      The third possibility is - nothing happens, when the given value is the same as the current 
127      one, or the _feed attribute is set to True (meaning the record is fed data via the feedDataRow 
128      function). 
129       
130      The C{setObjectID} method is an accessor to the _objectid attribute. 
131       
132         >>> aRecord._objectid = 1 
133         >>> aRecord.setObjectID(1) 
134       
135      are equivalent. 
136       
137      The C{objectid} attribute holds the current record value of the C{objectid} columns. Modifying 
138      it marks the record as I{modified} and the new objectid will be written to the database on the 
139      next C{write}. B{DO NOT DO THIS, IT IS EVIL, IT MAY KILL YOU}!
